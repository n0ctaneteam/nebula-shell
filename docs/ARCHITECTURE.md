# Nebula Shell Architecture

Version: 1.0.0

Status: Draft

---

# Purpose

This document defines the architectural rules of Nebula Shell.

The architecture is considered part of the source code.

If implementation conflicts with this document, this document is correct.

---

# Vision

Nebula Shell is a modern desktop shell framework for Wayland.

It provides a reactive runtime for building:

- Panels
- Bars
- Popups
- Launchers
- Notifications
- Dashboards
- Lock Screens
- Desktop Widgets
- Overlays

NebulaShell is **not** a desktop environment.

NebulaShell is **not** a window manager.

NebulaShell is **not** a GTK wrapper.

GTK is only the rendering backend.

---

# Core Principles

NebulaShell follows these principles.

1. Reactive

State changes should automatically propagate.

Widgets should never manually refresh themselves.

---

2. Event Driven

Prefer events.

Avoid polling.

---

3. Layered

Each layer has exactly one responsibility.

Dependencies only point downward.

Never upward.

---

4. Composable

Small components.

Small services.

Small widgets.

Never create giant objects.

---

5. Predictable

Public APIs should behave consistently.

No magic.

No hidden behavior.

---

6. AI Friendly

Architecture must be understandable by both humans and AI.

Consistency is more valuable than cleverness.

# Layer Responsibilities

## Core Runtime

Responsible for:

- application lifecycle
- object registry
- logging
- configuration
- plugin loading
- IPC bootstrap
- initialization
- shutdown

Core Runtime knows nothing about widgets.

---

## Window System

Responsible for:

- windows
- panels
- overlays
- popups
- monitor management
- anchors
- layers

No system state belongs here.

---

## Reactive System

Responsible for:

- properties
- bindings
- signals
- observers

No UI logic.

No GTK logic.

Only state synchronization.

---

## Services

Responsible for system state.

Examples:

Battery

Audio

Bluetooth

Media

Network

Workspace

Notification

Services own state.

Widgets never own state.

---

## Widgets

Widgets display information.

Widgets subscribe to services.

Widgets never communicate with DBus.

Widgets never spawn shell commands.

Widgets never perform filesystem access.

Widgets never perform networking.

---

## Python API

Python is the user interface of the framework.

Python should feel handwritten.

Not autogenerated.

Simple.

Readable.

Python wraps stable public APIs only.

# State Ownership

Every piece of state has one owner.

Example

Battery percentage

↓

BatteryService

Widget

↓

Read only

Widgets never duplicate state.

---

# Services

Services are singleton objects.

Example

BatteryService.default()

AudioService.default()

WorkspaceService.default()

Every service exposes:

Properties

Signals

Methods

No rendering.

---

# Reactive Model

Properties

↓

Signals

↓

Bindings

↓

Widgets

Widgets should never manually synchronize values.

Bindings perform synchronization automatically.

---

# Rendering

Rendering backend is private.

Current backend

GTK4

Future backends may exist.

The public API must not prevent replacing the renderer.

---

# Wayland

Wayland is mandatory.

X11 compatibility is intentionally not supported.

NebulaShell optimizes for Wayland first.

---

# Hyprland

Hyprland receives first-class support.

Compositor-specific integrations belong inside adapters.

Example

core/compositors/hyprland/

Never scatter Hyprland checks throughout the codebase.

---

# Plugins

Plugins extend the framework.

Plugins may provide:

Services

Widgets

Themes

Utilities

Plugins should not modify framework internals.

Plugins communicate through public APIs.

---

# Themes

Theme engine is independent.

Widgets expose style classes.

GTK CSS performs rendering.

Widgets should not contain hardcoded colors.

---

# Configuration

Configuration is Python.

Configuration constructs objects.

Configuration should not contain framework logic.

Keep configs declarative.

---

# Threading

Main Thread

GTK

Widgets

Window System

Worker Threads

IO

Parsing

Heavy computation

Never access GTK from worker threads.

---

# Memory

Prefer ownership over sharing.

Avoid global mutable state.

Prefer immutable values where practical.

---

# Error Handling

Recover whenever possible.

Never silently ignore errors.

Never terminate the runtime for recoverable failures.

---

# Performance Rules

No polling unless absolutely necessary.

Prefer signals.

Prefer event subscriptions.

Avoid unnecessary allocations.

Avoid unnecessary redraws.

Reuse objects where possible.

Initialize expensive objects lazily.

---

# File Layout

One public class per file.

One responsibility per class.

Avoid utility files containing unrelated code.

---

# Naming

Directories describe domains.

Examples

core/

services/

widgets/

animation/

ipc/

plugin/

Never organize by file type.

Organize by responsibility.

---

# Public API Stability

Everything inside the public API should remain stable.

Internal implementation may change freely.

Breaking changes require a major version.

---

# Future Expansion

NebulaShell should support future additions without rewriting the core.

Examples

new widgets

new services

new plugins

new animations

new compositors

new language bindings

Architecture should encourage extension rather than modification.

---

# Final Rule

When faced with multiple implementation choices:

Choose the one that

- keeps dependencies clean
- reduces coupling
- improves maintainability
- preserves API stability
- avoids exposing implementation details

Architecture is more important than convenience.
